# خطة تطوير الوكيل الذكي (Agent) — بدون تنفيذ

الهدف: نخلي الوكيل أذكى، أدق، أسرع، وأقل أخطاء، بدون ما نخرب الردود الحالية الشغالة.

---

## 1) ذاكرة الزبون (Customer Memory)
- جدول `customers` (telegram_user_id, name, phone, default_address, area, last_order_at, total_orders, lifetime_value, tags[]).
- يربط تلقائياً مع `conversations.customer_id`.
- الوكيل يقرأ الذاكرة بأداة جديدة `get_customer_profile` → يعرف الاسم، آخر طلب، العنوان المعتاد.
- فائدة: "أهلاً أبو علي، نفس العنوان مال المرة الماضية؟" بدل ما يسأل كل مرة.

## 2) أداة "إعادة طلب سابق" (Reorder)
- أداة `get_last_orders(limit)` ترجع آخر 3 طلبات للزبون.
- أداة `reorder(order_id)` تنسخ نفس الأصناف للسلة مباشرة.
- يقلل خطوات الطلب من 6-7 رسائل إلى 2.

## 3) فهم النية قبل التنفيذ (Intent Router)
- خطوة أولى خفيفة (Gemini Flash Lite) تصنّف الرسالة:
  `menu | order | status_check | complaint | working_hours | location | reorder | smalltalk | other`.
- حسب النية يختار أدوات أقل ويرد أسرع وأرخص.
- مهم: لو النية `status_check` يستدعي مباشرة `get_order_status` بدون لف.

## 4) متابعة حالة الطلب من الزبون
- أداة `get_order_status(order_id?)` — لو ما عنده رقم يجيب آخر طلب نشط.
- يرد بحالة مفهومة + الوقت المتوقع.

## 5) Upsell ذكي (مدروس، مو مزعج)
- بعد `add_to_cart` الوكيل يفحص `combo_suggestions` أو فئة "مشروبات/إضافات".
- يقترح مرة واحدة فقط بكل طلب (flag بـ conv.meta).
- قاعدة صارمة: ممنوع يقترح إذا الزبون قال "بس هذا" أو "خلص".

## 6) منع الأخطاء الشائعة (Guardrails)
- **سقف تكرار الأدوات**: نفس الأداة بنفس الـ args ما تنعاد بنفس الـ turn.
- **Timeout**: لو الموديل ما رد بـ 25 ثانية → رسالة "لحظة..." + handoff.
- **Loop detection**: إذا تجاوز 4 tool calls بدون نص نهائي → handoff تلقائي.
- **Schema validation للـ args** قبل تنفيذ الأداة (Zod).

## 7) تحسين البحث بالمنيو
- حالياً embedding + ilike. نضيف:
  - **مرادفات** (`menu_aliases` جدول): "بيبسي" = "كولا"، "صمون" = "خبز".
  - **تصحيح إملائي** خفيف (Levenshtein) قبل البحث.
  - **Fuzzy match** لو ما طلع نتيجة.

## 8) دعم الصور من الزبون (Vision)
- إذا الزبون دز صورة منيو/شاشة → نمررها للموديل (Gemini يدعم رؤية).
- مفيد لما يصور طلب صديقه أو إيصال.

## 9) أوقات العمل + جدولة طلبات
- موجود فحص "مفتوح/مغلق". نضيف:
  - أداة `schedule_order(time)` تخزن `scheduled_for` بالطلب.
  - تنبيه تلقائي للمطعم وقت اقتراب الموعد.

## 10) تعدد اللغات تلقائي
- كشف لغة الرسالة (عربي فصحى / عراقي / إنجليزي / كردي).
- يرد بنفس اللهجة بدون ما المطعم يضبط شي.

## 11) جودة الردود (Prompt Engineering)
- تقسيم الـ system prompt إلى:
  - **Identity** (من أنت)
  - **Rules** (ممنوعات)
  - **Style** (نبرة)
  - **Context** (مطعم، سلة، زبون)
  - **Examples** (3-4 أمثلة few-shot).
- إضافة أمثلة محادثات مثالية → يقلل الأخطاء 30-40%.

## 12) تقييم وتحسين مستمر (Eval Loop)
- جدول `agent_evaluations` (conversation_id, rating, issue_type, notes).
- بعد كل طلب مكتمل: زر بالداشبورد "هل الوكيل أحسن؟ 👍/👎".
- صفحة "محادثات تحتاج مراجعة" (handoffs + ratings سيئة).
- استخدمها لتحسين الـ prompt شهرياً.

## 13) Cache للأسئلة المتكررة
- "وين موقعكم؟" / "شنو دواماتكم؟" / "تجيبون توصيل؟" → جواب جاهز من DB، ما يروح للموديل.
- جدول `faq_cache` (restaurant_id, question_pattern, answer).
- يوفر ~40% من استدعاءات الموديل.

## 14) إشعارات للمطعم (Real-time Alerts)
- طلب جديد → Telegram للمطعم فوراً (موجود؟ نحسنه).
- handoff → تنبيه عاجل.
- زبون متضايق (sentiment سلبي) → تنبيه قبل ما يصير handoff.

## 15) Analytics للوكيل نفسه
- معدل نجاح الطلبات (محادثة → طلب مؤكد).
- متوسط عدد الرسائل لإكمال طلب.
- نسبة الـ handoff.
- التكلفة لكل طلب (tokens × سعر الموديل).
- الأدوات الأكثر استخداماً والأكثر فشلاً.

## مبادئ صارمة أثناء التنفيذ
1. **ما نلمس** `runTool` للأدوات الموجودة — بس نضيف جديدة.
2. كل أداة جديدة تترجم لـ `agent_logs` للتشخيص.
3. أي تغيير بالـ prompt → نختبر على 5 محادثات سابقة قبل الاعتماد.
4. Feature flags بجدول `restaurants.agent_features` (jsonb) — كل مطعم يفعّل اللي يريده.

---

**شنو تحب نبدي بيه؟** اقترح المرحلة **A (Quick Wins)** لأنها أسرع وتعطي فرق فوري بدون مخاطر.
