
# دعم الفروع المتعددة (Multi-Branch)

الفكرة: المطعم الواحد يكدر يضيف عدة فروع، وكل فرع إله منطقة توصيل وأوقات دوام وأرقام تواصل خاصة بيه. الزبون بالتلغرام يحجي بعنوانه، والبوت تلقائياً يوجه الطلب لأقرب فرع.

## 1) قاعدة البيانات

جدول جديد `branches`:
- `restaurant_id` (يربط بالمطعم الرئيسي)
- `name` (مثلاً "فرع الكرادة"، "فرع المنصور")
- `address` (عنوان الفرع نفسه)
- `phone` 
- `delivery_areas` (jsonb — قائمة المناطق اللي يخدمها)
- `open_hours` (jsonb — كل فرع ممكن إله دوام مختلف)
- `min_order`, `is_active`
- `telegram_chat_id` (اختياري — للإشعارات الخاصة بالفرع)

تحديثات:
- `orders` → نضيف عمود `branch_id` (اختياري للطلبات القديمة).
- `menu_items` → نخلي السعر والتوفر مشترك بين الفروع (مادام نموذج "حساب واحد + منيو موحد")، بس نضيف اختيارياً `unavailable_branch_ids` للأصناف اللي ما متوفرة بفرع معين.
- ترحيل البيانات الموجودة: ننقل `delivery_areas` و `open_hours` من `restaurants` لأول فرع افتراضي.

RLS: كل السياسات تبقى عبر `restaurant_id` (المالك يدير كل الفروع).

## 2) لوحة الإدارة (Dashboard)

تبويب جديد **"الفروع"**:
- بطاقات الفروع (إضافة/تعديل/حذف).
- لكل فرع: اسم، عنوان، رقم، مناطق التوصيل، أوقات الدوام، الحد الأدنى للطلب، تفعيل/إيقاف.

نظرة عامة + تفاصيل (حسب اختيارك):
- **صفحة "نظرة عامة"** بأعلى الداشبورد: إحصائيات مجمعة لكل الفروع (إجمالي طلبات اليوم، الإيرادات، الطلبات النشطة لكل فرع جنب بعض).
- **فلتر الفرع** بأعلى تبويب الطلبات: "كل الفروع" أو فرع معين.
- بكل طلب نعرض شارة (badge) باسم الفرع.

## 3) البوت الذكي (Telegram Agent)

أداة جديدة `resolve_branch(address)`:
- تحلل العنوان اللي يكتبه الزبون.
- تطابقه مع `delivery_areas` لكل فرع.
- ترجع أقرب فرع متوفر + أوقات دوامه.

تحديث `add_to_cart` و `create_order`:
- يحفظ `branch_id` بالطلب.
- يرفض الطلب إذا الفرع مغلق أو المنطقة ما مخدومة، ويقترح بديل.

تحديث system prompt للوكيل:
- يسأل عن العنوان من البداية إذا الزبون يريد توصيل.
- يخبر الزبون بأي فرع راح يوصله ووقت التوصيل المتوقع.

## 4) خطوات التنفيذ بالترتيب

1. **Migration**: إنشاء جدول `branches`، إضافة `branch_id` للطلبات، نقل البيانات الموجودة.
2. **Dashboard**: تبويب "الفروع" + CRUD كامل.
3. **نظرة عامة**: صفحة/قسم إحصائيات مجمعة + فلتر بالطلبات.
4. **Agent**: أداة `resolve_branch` + تحديث منطق الطلبات + system prompt.
5. **اختبار**: محادثة كاملة بمنطقتين مختلفتين للتأكد من التوجيه.

## تفاصيل تقنية

- مطابقة المناطق: نبدأ بمطابقة نصية بسيطة (ilike + قائمة المناطق)، ولاحقاً نقدر نضيف Gemini Flash Lite يستخرج المنطقة من العنوان الحر.
- اختيار الفرع لما المنطقة مخدومة من فرعين: نأخذ الأقل ضغطاً (عدد طلبات نشطة) أو الأول حسب الترتيب.
- التوافق مع الطلبات القديمة: `branch_id` nullable، الواجهة تتعامل مع `null` كـ"غير محدد".
