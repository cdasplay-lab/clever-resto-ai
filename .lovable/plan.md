# المرحلة 8 — ثلاث قفزات تخلّي الوكيل بمستوى عالمي

## الهدف
بعد ما البوت صار يبيع ويفهم السلة والمخزون والكومبوهات، ناقصنا ثلاث قدرات أساسية موجودة بأقوى بوتات الطلبات بالعالم:

1. **فهم الرسائل الصوتية** — الزبون يحجي بصوته، الوكيل يفهم ويرد.
2. **طلبات مجدولة (لوقت لاحق)** — "أريد الغداء الساعة 2" يحجزه ويرسله للفرع بالوقت المحدد.
3. **تسليم ذكي للموظف (Human Handoff)** — لما البوت يحتار 3 مرات، أو الزبون يطلب موظف، يعلّم المحادثة ويرسل إشعار للمالك بدل ما يلف بالحجي.

---

## 1) رسائل صوتية (Voice notes)

### قاعدة البيانات
بدون تغييرات. الصوت يتحول لنص قبل أن يدخل التاريخ.

### `telegram-webhook/index.ts`
- لما يجي `message.voice` أو `message.audio`:
  - نزّل الملف عبر `getFile` من تيليجرام.
  - أرسله إلى Lovable AI (`google/gemini-2.5-flash` يدعم الصوت) أو OpenAI Whisper عبر LOVABLE_API_KEY.
  - النص الناتج يُمرر للوكيل كـ user message عادي.
  - أضف بادئة قصيرة بالرسالة المخزنة: `[رسالة صوتية] ...النص...` حتى المالك يميز بسجل المحادثات.

### ملاحظات
- لو فشل التحويل، يرد البوت: "ما كدرت أفهم الصوت، ممكن تكتبلي؟" بدل ما يتعطل.
- يشتغل لتيليجرام أولاً (واتساب وانستا بمراحل لاحقة).

---

## 2) طلبات مجدولة

### قاعدة البيانات (Migration)
- `orders.scheduled_for timestamptz nullable` — لو NULL يعني الآن.
- `orders.status` enum يبقى نفسه؛ نضيف فلتر "مجدولة" بالواجهة عبر `WHERE scheduled_for > now()`.
- دالة `cron` (pg_cron) كل دقيقة:
  - تجلب الطلبات `status='scheduled'` و`scheduled_for <= now() + interval '30 min'` (نرسلها للفرع قبل نص ساعة من الموعد).
  - تستدعي edge function `notify-order-status` لإرسالها لتيلكرام الفرع وتحديث الحالة.

### `agent-run/index.ts`
- أداة جديدة **`schedule_order`** بنفس بارامترات `preview_order`/`submit_order` + `scheduled_time` (نص حر: "بكرة الساعة 2"، "الساعة 7 مساءً"). الأداة:
  - تحول النص لـ ISO timestamp بمنطقة بغداد (Asia/Baghdad) عبر منطق بسيط داخل الأداة (regex لساعات + كلمات: "بعد ساعة"، "بكرة"، "اليوم").
  - تتحقق إن الموعد > now() + 15 دقيقة وضمن ساعات عمل الفرع.
  - تنشئ الطلب بـ `status='scheduled'` و`scheduled_for=...`.
- قاعدة بالـ system prompt: لما الزبون يذكر وقت ("بكرة"، "الساعة"، "بعد ساعة")، اسأل عن الوقت الدقيق ثم استدعِ `schedule_order` بدل `submit_order`.

### الواجهة (`dashboard.tsx` — OrdersTab)
- تبويب فرعي "مجدولة" يعرض الطلبات بـ `scheduled_for` مع عدّاد تنازلي.
- شارة "مجدول" بطاقة الطلب + الوقت.

---

## 3) تسليم ذكي للموظف (Human Handoff)

### قاعدة البيانات
- بدون تغييرات (نستغل `conversations.is_bot_paused` الموجود + `assigned_to`).

### `agent-run/index.ts`
- أداة جديدة **`request_human`** يستدعيها الوكيل عند:
  - الزبون يطلب صراحة: "أريد أحجي وية واحد"، "محتاج موظف"، "ما فهمت".
  - الوكيل فشل يفهم 3 رسائل متتالية (يتتبع `meta.confusion_count`).
  - الزبون يكرر شكوى أو يستخدم كلمات غضب.
- الأداة:
  - تضع `is_bot_paused=true` بالمحادثة.
  - ترسل إشعار تيليجرام للمالك (chat_id من `restaurants.meta.owner_telegram` لو متوفر، أو لكل الفروع المفعلة): "🧑‍💼 [الزبون] يحتاج موظف بمحادثة [link]. آخر رسالة: ...".
  - ترد على الزبون: "حوّلت محادثتك لزميل بشري راح يجاوبك خلال دقائق 🙏".
- قاعدة بالـ system prompt: لا تكرر نفس السؤال مرتين. لو حسيت بالتوتر أو طلب صريح، استدعِ `request_human` فوراً.

### الواجهة
- `customers-tab.tsx`: زر "استلام المحادثة" يفك الإيقاف ويعيد البوت للعمل.
- مؤشر بصري بالمحادثات المعلّقة (Badge أحمر "ينتظر موظف").

# المرحلة 10 — تحليلات وتقارير متقدمة ✅

تبويب "التحليلات" أعيدت كتابته بالكامل بمكوّن مستقل `src/components/analytics-tab.tsx`:

- **نطاق زمني**: 7 / 30 / 90 يوم.
- **مؤشرات (KPIs)**: الطلبات، الإيرادات، متوسط الطلب، المحادثات، نسبة التحويل (طلبات/محادثات)، الزبائن الفريدون، نسبة العودة، تشغيلات البوت.

- **قوائم**: الأكثر طلباً (مع الإيراد لكل صنف)، أفضل الزبائن (حسب الإنفاق).
- **تصدير CSV** للبيانات اليومية.

الملفات: `src/components/analytics-tab.tsx` (جديد)، `src/routes/dashboard.tsx` (استبدال التبويب القديم).
